# Contributing Guide

## Prerequisites

- [Terraform](https://developer.hashicorp.com/terraform/install) >= 1.4.0
- [pre-commit](https://pre-commit.com/#install)
- [terraform-docs](https://terraform-docs.io/user-guide/installation/)
- AWS credentials configured locally

## Setup

Clone the repo and install the pre-commit hooks:

```bash
git clone https://github.com/YOUR_ORG/terraform-aws-eks.git
cd terraform-aws-eks
pre-commit install
```

The hooks will automatically run `terraform fmt`, update `README.md` docs, and check for merge conflicts and whitespace issues on every commit.

## Branching

Use short, descriptive branch names prefixed by type:

| Prefix | When to use |
|--------|-------------|
| `feat/` | New feature or variable |
| `fix/` | Bug fix |
| `docs/` | Documentation only |
| `refactor/` | Code restructure, no behavior change |
| `chore/` | Dependency bumps, tooling |

```bash
git checkout -b feat/my-feature
```

## Adding a Feature

### 1. Add the variable

Define it in `variables.tf` with a clear description and a safe default so existing callers are not broken:

```hcl
variable "my_feature" {
  description = "(Optional) ..."
  type        = ...
  default     = {}
}
```

### 2. Implement the resource

Add the resource or data source in `main.tf`. Use `for_each` over maps to keep resources dynamic and avoid index-based drift.

### 3. Expose outputs

Add relevant outputs in `outputs.tf`:

```hcl
output "my_feature" {
  description = "..."
  value       = aws_some_resource.self
}
```

### 4. Update the examples

Add the new variable to at least one example under `examples/` so users can see a working configuration. Use placeholder ARNs (`123456789012`) for anything account-specific.

### 5. Update the changelog

Add an entry under a new `[Unreleased]` section at the top of `CHANGELOG.md`:

```markdown
## [Unreleased]

### Added
- Brief description of what was added
```

### 6. Commit

Follow [Conventional Commits](https://www.conventionalcommits.org/):

```
feat: add support for X
fix: correct Y behavior
docs: update README for Z
```

Pre-commit will auto-format Terraform files and regenerate the `README.md` inputs/outputs table on commit.

## Pull Request Checklist

Before opening a PR make sure:

- [ ] `terraform fmt` passes (enforced by pre-commit)
- [ ] `README.md` inputs/outputs table is up to date (auto-generated by pre-commit)
- [ ] At least one example reflects the new feature
- [ ] `CHANGELOG.md` has an `[Unreleased]` entry
- [ ] No hardcoded account IDs, regions, or secrets

---

## Release Process

Releases follow [Semantic Versioning](https://semver.org/):

| Change type | Version bump |
|-------------|-------------|
| Backwards-compatible new feature | `MINOR` (1.x.0) |
| Bug fix, no API change | `PATCH` (1.0.x) |
| Breaking change | `MAJOR` (x.0.0) |

### Steps

**1. Finalize the changelog**

Rename `[Unreleased]` to the new version and date, and add a comparison link at the bottom:

```markdown
## [1.2.0] - 2026-05-01
```

```markdown
[1.2.0]: https://github.com/YOUR_ORG/terraform-aws-eks/compare/v1.1.0...v1.2.0
```

**2. Commit**

```bash
git add CHANGELOG.md
git commit -m "docs: update changelog for v1.2.0"
```

**3. Tag**

```bash
git tag -a v1.2.0 -m "Release v1.2.0"
```

**4. Push**

```bash
git push origin master --tags
```

**5. GitHub Release**

- Go to **Releases → Draft a new release**
- Select tag `v1.2.0`
- Paste the changelog entry as release notes
- Publish

The Terraform Registry will pick up the new tag automatically if the repo is already connected.
